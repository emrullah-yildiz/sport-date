Token-efficiency policy (adopted 2026-07-04, owner-directed) — cut cost, keep quality:

1 · Right-size the model per task. Mechanical work (rendering, status edits, file moves) runs on the small/cheap model tier; full reasoning is reserved for building, verifying, and strategy. Quality lives in the hard steps — not in using a big model to rename files.
2 · State lives in files, not in conversation. HQ, tickets, and logs carry the memory; long chat context is the single biggest token burner. Fresh sessions per workstream start from these files instead of dragging history.
3 · Agents return conclusions, not file dumps. Background agents read narrowly (grep/sections, never whole trees) and report a summary; their full transcripts never enter my context.
4 · Batch, don't drip. One wake-up produces a whole batch (e.g. all creatives for the week) instead of many small turns that each re-read the growing context.
5 · No polling, no re-verification. Event-driven wake-ups only (agent-completion notifications); green tests/builds are trusted once, not re-read.
6 · Template reuse. The creative system is templated — a new post is a small text delta on a proven design, not a fresh design exploration.
7 · Tight tickets. Precise acceptance criteria stop agents from wandering; wandering is where quality-neutral tokens go to die.
Quality floor (non-negotiable): full verification (typecheck/lint/test/prod-build), adversarial review on revenue-path code, and full-reasoning strategy stay at maximum — savings come from cheap steps, never from the definition of done.
