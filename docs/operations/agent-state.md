# Agent state

## Current outcome

Fixed the systemic user-deletion / audit-immutability conflict that blocked GDPR right-to-erasure of any `users` row. The four append-only audit tables (`moderation_audit_log`, `moderation_case_access_log`, `moderation_evidence_references`, `role_audit_log`) each paired an `ON DELETE SET NULL` FK to `users` with an `ON UPDATE … DO INSTEAD NOTHING` rule, so the FK's internal SET-NULL UPDATE was swallowed and Postgres raised "referential integrity query … gave unexpected result" — no user could be hard-deleted. New migration `apps/web/db/019_audit_append_only_allows_user_nulling.sql` drops each `no_update` rule and replaces it with a `BEFORE UPDATE` trigger backed by one shared PL/pgSQL function (`audit_append_only_allow_user_nulling`, parameterised by the user-ref column names via `TG_ARGV`). The trigger compares `OLD`/`NEW` as `jsonb`: any change to a non-user-ref column raises, and a user-ref column may change only from NON-NULL to NULL (the exact shape of the FK SET-NULL action) — setting a user-ref column to a non-null value also raises. The four `no_delete` rules are kept, so audit rows remain app-undeletable; only the system FK SET-NULL is newly permitted. The migration runner splits on `;`+newline, which would shred a multi-line `$$`-quoted body, so the `CREATE FUNCTION` body is authored on a single physical line (no internal `;` ends a line); verified by simulating the split and by applying `019` cleanly to the dev DB. Proven against real SQL with opt-in tests in `apps/web/src/lib/audit-erasure.integration.test.ts` (2 tests, `RUN_DB_INTEGRATION=1`, sentinel rows, full cleanup) for both `moderation_audit_log` and `role_audit_log`: (a) a referenced user is now hard-deletable, the audit row SURVIVES with the user-ref column NULL and every other column unchanged (incl. `created_at` and `metadata`); (b) an application UPDATE of a non-user-ref column is still REJECTED (and a user-ref → non-null UPDATE is rejected); (c) an application DELETE of an audit row is still a no-op (row remains). The other two tables share the identical trigger/function and surviving `no_delete` rule and are covered by inference (all four installs verified in `pg_trigger`/`pg_rewrite`). Dev DB left pristine (all audit tables 0 rows, no sentinel residue, `registrations` untouched); only the pre-existing auth-email fixture user remains. Default suite stays hermetic at 156 (99 web + 57 domain; the 2 new integration tests self-skip among the 10 skipped); +2 opt-in DB tests (10 total opt-in); typecheck green.
