## 2026-06-29 - Resolve audit append-only immutability vs ON DELETE SET NULL with a per-column BEFORE UPDATE trigger (migration 019)

Fixed the systemic GDPR right-to-erasure blocker: no `users` row could be hard-deleted because four append-only audit tables (`moderation_audit_log`, `moderation_case_access_log`, `moderation_evidence_references`, `role_audit_log`) each paired an `ON DELETE SET NULL` FK to `users` with an `ON UPDATE … DO INSTEAD NOTHING` rule, so the FK's internal SET-NULL UPDATE was swallowed and Postgres raised "referential integrity query … gave unexpected result."

**Mechanism chosen.** New migration `apps/web/db/019_audit_append_only_allows_user_nulling.sql` drops each `no_update` rule and replaces it with a `BEFORE UPDATE` trigger backed by ONE shared PL/pgSQL function, `audit_append_only_allow_user_nulling()`, parameterised per table by the user-reference column name(s) via `TG_ARGV` (`actor_user_id`; `target_user_id`+`actor_user_id` for `role_audit_log`; `created_by_user_id` for evidence). The function compares `to_jsonb(OLD)` and `to_jsonb(NEW)` key by key: if a non-user-ref column changed it raises; if a user-ref column changed it is allowed ONLY when it went from a non-null value to NULL (`old <> 'null'::jsonb AND new = 'null'::jsonb`), which is exactly the shape of the FK's SET-NULL action — setting a user-ref column to a non-null value, or any other edit, still raises (`restrict_violation`). This is how the trigger distinguishes the referential-integrity SET-NULL from an application edit: not by session/role, but by the precise column-set and null-direction of the diff. The four `ON DELETE … DO INSTEAD NOTHING` rules are deliberately KEPT, so audit rows remain undeletable by the application; the FKs never delete audit rows, they only null the reference. Chose the diff-shape trigger over (a) privilege/role-based immutability (no separate DB role exists here, and the FK action runs as the same role as the app) and (b) per-table bespoke functions (one `TG_ARGV`-driven function is less code and equally readable). Net effect: audit immutability against application writes/deletes is preserved exactly; only the system FK SET-NULL is newly permitted.

**Migration runner / `$$` handling.** `scripts/migrate.mjs` splits each file on `/;\s*(?:\r?\n|$)/`, i.e. a `;` that is the last non-space character before a newline ends a statement. A normal multi-line `$$ … $$` PL/pgSQL body would therefore be shredded (every internal `RAISE …;`, `END IF;`, `END LOOP;` would split). Verified this by simulating the exact split in Node, watching the body fragment into 8 pieces. Worked around it WITHOUT changing the runner by authoring the entire `CREATE OR REPLACE FUNCTION … END $$ LANGUAGE plpgsql` body on a SINGLE physical line, so no internal `;` is ever followed only by whitespace+newline; the split then yields one clean function statement plus the four `DROP RULE`/`CREATE TRIGGER` pairs (9 statements total). Confirmed by actually running `npm run db:migrate -w @sport-date/web` against the dev DB — `019` applied cleanly — and by inspecting `pg_rewrite`/`pg_trigger` (the four `no_update` rules gone, the four `no_delete` rules and four `*_append_only` triggers present, the function present).

**Two properties proven against real SQL.** Added opt-in tests `apps/web/src/lib/audit-erasure.integration.test.ts` (gated on `RUN_DB_INTEGRATION=1`, sentinel rows, full cleanup; runner extended to include the file) for both `moderation_audit_log` and `role_audit_log`: (a) ERASURE WORKS — a user referenced by an audit row is hard-deletable, the audit row survives with the user-ref column now NULL and every other column (incl. `created_at`, `metadata`, `reason`) unchanged; (b) IMMUTABILITY PRESERVED — an application UPDATE of a non-user-ref column still raises, and a user-ref → non-null UPDATE still raises; (c) an application DELETE of an audit row is still a no-op (the surviving `no_delete` rule). The other two tables (`moderation_case_access_log.actor_user_id`, `moderation_evidence_references.created_by_user_id`) carry the identical shared trigger/function and surviving `no_delete` rule and are covered by inference (all four installs verified in the catalogs) — a direct test for them is the next integration-coverage outcome. Cleanup deletes audit rows by their own primary key (their user columns get nulled by the fix, so a by-user-id delete would strand them) via a transient `DISABLE RULE … no_delete`/`ENABLE RULE`; dev DB left pristine (all audit tables 0 rows, no sentinel residue, `registrations` untouched). Default suite stays hermetic at 156 (99 web + 57 domain; the 2 new tests self-skip among the 10 skipped); +2 opt-in DB tests (10 opt-in total); typecheck green.

Residual risk stated honestly: two of the four tables are proven by inference, not directly; and the test asserts the property at the audit-table boundary (a direct `DELETE FROM users`), not yet through the application's full account-deletion code path. ## 2026-06-29 - Make "never reward prohibited signals" a regression-proof tested guard (tests only, no rule change)

Honest finding about how progression is computed: the only pure-domain progression function, `calculateMovementProgress(attendedMoves: number)`, takes a *single non-negative integer*. None of the prohibited signals are even representable as inputs to it — there is no swipe, view, skip, decline, report, streak, or duration parameter anywhere in the progression math. What the guard asserts, per prohibited signal: (1) swipes/views — browsing yields no reflection, so baseline+0 == baseline; (2) skip decisions — driving `skipJoinRequest` up to the boundary changes a join-request, not the arc; (3) rejection/decline — auto-decline at max skips earns no move; (4) report suppression — a report (filed or withheld) is not a progression input at all; (5) streaks/consecutive days — calendar time is not attendance; (6) time/session duration — session seconds are not attendance. ## 2026-06-29 - Real-SQL integration tests are opt-in, non-destructive, and surfaced two bugs

Gate 7 was unblocked with the owner-provided sport-date dev database. Run the 18 additive migrations to bring a database to current schema, then exercise the real `auth-email` functions against Postgres rather than mocks. Keep these tests opt-in (`RUN_DB_INTEGRATION=1`, `npm run test:integration`) so the default suite stays hermetic and CI-safe, and non-destructive: reuse one fixed sentinel fixture user (`int-test+fixture@sport-date.invalid`) with per-test child-row cleanup; never read or modify other rows or the `registrations` table. A dedicated disposable/CI Postgres remains preferable to the shared dev DB.

The tests found two real defects the mocks hid. (1) `issueEmailVerificationTokenForUser` raised `invalid input syntax for type uuid: ""` whenever no row was inserted (already-verified or non-active user), because `COALESCE((SELECT id FROM inserted), '')` coerces the empty-string fallback to `uuid`; fixed by selecting `id::text`. (2) No `users` row can be hard-deleted: four append-only audit tables (`moderation_audit_log`, `moderation_case_access_log`, `moderation_evidence_references`, `role_audit_log`) combine an `ON DELETE SET NULL` FK to `users` with an `ON UPDATE … DO INSTEAD NOTHING` immutability rule, so the FK's internal SET-NULL UPDATE is swallowed and Postgres raises "referential integrity query … gave unexpected result." This blocks GDPR right-to-erasure of the row; soft-delete has masked it in production. Deliberately NOT fixed in the same commit — it changes a moderation/role audit-immutability control and must be a separate, reviewed migration that allows only the FK's SET-NULL transition while keeping member/moderator writes blocked. Test residue was removed by transiently dropping the affected FK constraints (audit tables empty), deleting, and restoring each from its captured `pg_get_constraintdef`, via the direct (non-pooler) endpoint to avoid cached RI plans.
