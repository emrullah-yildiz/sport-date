## 2026-06-30 - Split readiness from liveness (`/api/health/ready`) and commit to forward-only migrations in the deployment runbook

Two operational-prep decisions landed alongside the in-progress first Vercel deploy.

**Readiness is a separate endpoint from liveness — not a flag on `/api/health`.** The existing `/api/health` is a pure liveness probe (no DB/provider call) so it answers `200 { status: "ok" }` even before the stack is wired and is safe for an external uptime monitor. A dependency check needs the opposite property (it MUST touch the database), so conflating them would make one of the two behaviors wrong. New `apps/web/src/app/api/health/ready/route.ts` performs ONE cheap `SELECT 1` via `getDatabase()` and returns `200 { status: "ready" }` or `503 { status: "not_ready", reason }`. The deliberate security choice: the body carries only a **coarse, fixed `reason`** (`"database_not_configured"` for `DatabaseNotConfiguredError`, `"database_unreachable"` for any other client/query error) and **never the underlying error message** — a failed `SELECT 1` or a failed client construction can surface a driver error whose text embeds the host/connection string/credentials, so those are swallowed and never echoed. Same route conventions as the health/cron routes (`runtime="nodejs"`, `dynamic="force-dynamic"`, `Cache-Control: no-store` — a stale cached "ready" would hide a DB outage). This gives a monitor/orchestrator a way to distinguish "process up but DB unreachable" (503 here, 200 on liveness) from a full outage. Test mirrors the cron route test (`server-only` + `@/lib/db` mocked): 200 on resolve, 503 on `DatabaseNotConfiguredError`, 503 on query reject, no-store, and an explicit assertion that no response body contains the connection string on either failure path. Web suite 117 -> 122; full hermetic 174 -> 179; opt-in DB unchanged at 12; typecheck green; production build still compiles (`/api/health/ready` is a dynamic route).

**Migrations are forward-only; rollback is Vercel instant app-rollback, not down-migrations.** Recorded in `docs/operations/deployment-runbook.md`: there are intentionally **no down-migrations** in this repo. Migrations are additive (`… IF NOT EXISTS`, additive `ALTER`s) and written so an older app version keeps working against a newer schema — which is exactly what makes Vercel's instant rollback (promote a previous deployment) safe even after a migration has run. Rationale: hand-written down-migrations are an extra failure mode and a data-loss risk; at pre-launch a forward-only, additive discipline is simpler and safer than maintaining reversible pairs. A bad schema change is corrected with a NEW forward migration (`019` is the model), never by editing/deleting an applied migration or its `schema_migrations` row; a (disallowed) destructive migration is treated as an incident and recovered via Neon PITR into a branch. The runbook is concrete to THIS stack (Neon EU + Vercel `fra1` + Upstash + Sentry) and repeats the migration-runner caveat from the migration-019 entry (the `;`+newline split shreds multi-line `$$` bodies → author function bodies on one physical line). ## 2026-06-30 - Gate 6 IMPLEMENTED: async rate-limit store + env-gated Upstash REST adapter with in-memory graceful degradation

Completed Gate 6 (shared/distributed rate limiting) by making the rate-limit store and its public check functions async end-to-end and dropping in an env-gated Upstash Redis REST adapter behind the existing `RateLimitStore` seam, mirroring the disabled-by-default email-delivery adapter. **No new npm dependency** (`fetch` only). **Async change + behavior preservation.** Redefined the `RateLimitStore` interface around the two operations the existing two-phase consume logic actually needs — `peek(key, nowMs)` (read the current window) and `bump(key, nowMs, windowMs)` (atomic create-or-increment) — plus `sweepExpired`/`clear`, all returning Promises. `consumeRateLimit` keeps its EXACT semantics: a read phase peeks every rule and decides the block (max retry-after; a blocked request increments NOTHING), then a commit phase bumps every rule only when NOT blocked. **Upstash adapter.** `createUpstashRateLimitStore(url, token, fallback, fetchImpl?)`, activated by `resolveRateLimitStore` ONLY when BOTH `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are set. Talks to Upstash's REST **pipeline** endpoint (`POST {url}/pipeline`, `Authorization: Bearer {token}`, JSON array of command arrays). `bump` = one atomic pipeline `[["INCR", key], ["PEXPIRE", key, windowMs]]` (INCR creates the key at 1 or increments the live window; PEXPIRE arms/keeps the fixed-window expiry); `peek` = `[["GET", key], ["PTTL", key]]`, reconstructing `{ count, resetAt: nowMs + ttl }` and treating PTTL ≤ 0 (missing/no-expiry) as an absent window. It deliberately **fails to in-memory enforcement, never to unlimited**: an Upstash outage degrades shared cross-replica enforcement back to the per-process baseline we already shipped, but abuse protection is never silently disabled. (Chosen over fail-open, which would remove the limiter during an outage — the exact moment abuse is most likely.)

## 2026-06-29 - Resolve audit append-only immutability vs ON DELETE SET NULL with a per-column BEFORE UPDATE trigger (migration 019)

Fixed the systemic GDPR right-to-erasure blocker: no `users` row could be hard-deleted because four append-only audit tables (`moderation_audit_log`, `moderation_case_access_log`, `moderation_evidence_references`, `role_audit_log`) each paired an `ON DELETE SET NULL` FK to `users` with an `ON UPDATE … DO INSTEAD NOTHING` rule, so the FK's internal SET-NULL UPDATE was swallowed and Postgres raised "referential integrity query … gave unexpected result."

**Mechanism chosen.** New migration `apps/web/db/019_audit_append_only_allows_user_nulling.sql` drops each `no_update` rule and replaces it with a `BEFORE UPDATE` trigger backed by ONE shared PL/pgSQL function, `audit_append_only_allow_user_nulling()`, parameterised per table by the user-reference column name(s) via `TG_ARGV` (`actor_user_id`; `target_user_id`+`actor_user_id` for `role_audit_log`; `created_by_user_id` for evidence). The function compares `to_jsonb(OLD)` and `to_jsonb(NEW)` key by key: if a non-user-ref column changed it raises; if a user-ref column changed it is allowed ONLY when it went from a non-null value to NULL (`old <> 'null'::jsonb AND new = 'null'::jsonb`), which is exactly the shape of the FK's SET-NULL action — setting a user-ref column to a non-null value, or any other edit, still raises (`restrict_violation`). This is how the trigger distinguishes the referential-integrity SET-NULL from an application edit: not by session/role, but by the precise column-set and null-direction of the diff. The four `ON DELETE … DO INSTEAD NOTHING` rules are deliberately KEPT, so audit rows remain undeletable by the application; the FKs never delete audit rows, they only null the reference. Chose the diff-shape trigger over (a) privilege/role-based immutability (no separate DB role exists here, and the FK action runs as the same role as the app) and (b) per-table bespoke functions (one `TG_ARGV`-driven function is less code and equally readable). Net effect: audit immutability against application writes/deletes is preserved exactly; only the system FK SET-NULL is newly permitted.
